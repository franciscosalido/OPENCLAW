# ADR-0002: OpenAI-Compatible Embeddings as the Internal Contract

**Status:** Accepted
**Date:** 2026-04-29

---

## Context

OpenClaw / Quimera requires a stable embeddings interface across local and
future gateway-capable paths.

The project already uses LiteLLM as the compatibility layer for model routing,
and the architecture benefits from keeping one internal contract for embeddings
regardless of the actual backend provider.

The system must support:

- local-first embeddings
- future backend substitution without rewriting callers
- consistent testing and error handling
- strict separation between interface contract and embedding model choice
- explicit vector metadata so different embedding spaces are never mixed

## Decision

The project standardizes the **internal embeddings contract** on the
OpenAI-compatible `/v1/embeddings` interface.

This means:

- new gateway-based embedding clients must target `/v1/embeddings`
- all embedding responses must be parsed using the OpenAI-compatible shape
- LiteLLM is the mandatory compatibility layer for routing embedding requests
- Ollama is the initial local embedding backend behind LiteLLM
- the current local embedding model remains `nomic-embed-text`
- future embedding backends may change, but only behind the same internal
  contract
- `quimera_embed` is the canonical application-facing embedding alias
- `local_embed` remains as a backward-compatible alias during the transition
- the existing direct `OllamaEmbedder` remains the active RAG embedder until a
  future migration PR preserves or replaces its retry/backoff/concurrency
  behavior

## Normative Statement

**OpenAI-compatible `/v1/embeddings` is the internal contract; Ollama local is
the initial backend; LiteLLM is the compatibility layer; the current concrete
local embedding model remains `nomic-embed-text` behind a semantic alias; any
future embedding model or provider change requires explicit
reembedding/reindexing. Embedding model identity must be hidden from
application callers but preserved in collection metadata.**

## Why This Decision Was Made

This decision gives the project uniform client behavior:

- one `EmbedClient`
- one response parser
- one timeout/error contract
- one test surface
- one migration boundary between callers and backend providers

This reduces coupling and allows backend substitution later without rewriting
the caller layer.

## Architecture Rules

### Internal embeddings contract

Standardize on `/v1/embeddings` OpenAI-compatible.

### Semantic aliases

Application-facing code should prefer `quimera_embed`.

`local_embed` may remain available as a compatibility alias until all scripts,
docs, tests, and downstream callers are migrated.

Concrete model names such as `nomic-embed-text` must not be used by
application callers. They may appear only in:

- LiteLLM configuration
- provider/backend metadata
- vector collection metadata
- migration and audit documentation
- operational docs that explain local setup

### Current embeddings backend

Use local Ollama through LiteLLM, keeping `nomic-embed-text` as the current
embedding model.

The current production RAG embedding path remains direct Ollama until a later
migration PR changes it explicitly.

### Remote chat models

Claude and OpenAI may coexist in the gateway as chat/reasoning providers
without changing the embeddings contract.

### Anthropic embeddings

Do **not** define "Anthropic embeddings" as a project backend. Anthropic does
not provide its own embedding model.

Anthropic may remain a remote chat/reasoning provider in a future gateway
sprint, but it must not be described as a native embeddings provider. If the
project uses Voyage or another embeddings provider in the future, it must be
represented as its own embedding provider and must require explicit collection
reindexing.

### Vector collection rule

Never mix embeddings generated by different models or providers in the same
vector collection.

Every vector collection must persist at least:

- `embedding_provider`
- `embedding_model`
- `embedding_dimensions`
- `embedding_version`

Recommended metadata:

- `embedding_contract`
- `embedding_alias`
- `embedding_backend`
- `created_at`
- `source_corpus_id`
- `reindex_required_on_model_change`

## Consequences

### Positive

- stable interface across local and future backends
- lower caller complexity
- easier testing and mocking
- safer migration path
- clearer gateway boundary between local and remote systems

### Negative

- OpenAI-compatible interface does not mean semantic interchangeability of
  vectors
- a backend swap still requires explicit collection migration
- additional discipline is required around metadata and reindexing

## Enforcement Rules

The following rules are mandatory:

1. New gateway-based embedding clients must consume `/v1/embeddings`, not
   backend-specific embedding endpoints.
2. LiteLLM must sit between future application callers and the embedding
   backend.
3. The initial backend remains local Ollama with `nomic-embed-text` until an
   explicit migration ADR or PR changes it.
4. `quimera_embed` is the canonical semantic embedding alias.
5. `local_embed` remains as a compatibility alias while transition work is
   active.
6. Every vector collection must persist at least:
   - `embedding_provider`
   - `embedding_model`
   - `embedding_dimensions`
   - `embedding_version`
7. If `embedding_model` or `embedding_provider` changes, the collection must be
   re-embedded explicitly.
8. Mixing vectors from different embedding models in the same collection is
   forbidden.
9. Anthropic may be used for remote chat generation, but not as a native
   embeddings provider.

## Out of Scope

This ADR does not:

- select a future remote embeddings provider
- authorize mixing collections
- migrate production embeddings immediately
- change Level 0 local-only restrictions
- reindex Qdrant
- modify existing vector collections
- change RAG retrieval behavior
- change chunking
- change prompt building
- introduce FastAPI, MCP, remote embedding providers, or quant tools

## Revision Policy

This ADR may be amended if the project changes the internal compatibility
contract or adopts a different gateway standard.

Any amendment must preserve explicit migration rules and must not silently
invalidate old vector collections. Amendments must be made in writing.
